0x00 前言

在西电，每天分散在各个平台的校园通知像海量的信息流一样轰炸着我们的精力。为了不错过关键的考试、放假和讲座，同时干掉无意义的广告流，我决定动手开发一个全自动的情报聚合 Agent —— Academic-information-report-agent。

本篇笔记将硬核复盘这个项目从最初的脚本拼凑，历经纯脚本到结合AI评判的固定流程到命令行中由LLM自主决定工具调用的agent，最终演进为具备“动态偏好注入”与“Gradio 自动化面板”2.0 架构的完整全过程。

0x01 第一阶段：数据源抓取与会话管理 (Scraper & Session)

整个系统的地基是数据采集。我们主要面临两个战场：超星学习通（ChaoXing）与微信公众号（WeChat）。

1. 动态凭证的“无头浏览器”劫持

超星和微信的 Web 端都有极其严格的鉴权机制。为了解决 Cookie 经常失效的问题，我设计了双通道会话捕获器：

  • SessionHarvester & WechatHarvester：当爬虫返回 401 Unauthorized 错误或本地缓存失效时，自动调起有头浏览器（Selenium/Playwright）提示人工扫码登录，捕获最新的 Token 和 Cookie 集合并将其持久化到 configs/wechat_auth.json 中。

0x03 第三阶段：网页初期构建

在引入定时任务（BackgroundScheduler）和 Gradio 界面后，新的灵异现象发生了：点击仪表盘时网页直接卡死，但 PowerShell 控制台一片死寂，没有任何报错。

核心漏洞分析

Gradio 默认在处理 UI 事件时是同步且多线程的，而 Python 自带的 SQLite 默认是单线程排他锁。
当后台定时任务正在往数据库批量写入抓取到的新情报时，主线程点击仪表盘触发了查询请求。由于 database.py 里的连接没有设置超时，主线程进入了无休止的“死等”状态。因为是线程挂起而非代码异常，系统不会抛出任何 Traceback！

0x04 第四阶段：Gradio 驾驶舱建设与去 Emoji 视觉整改

一个合格的工具需要配上优雅的“驾驶舱”。我们基于 Gradio 搭建了全新的三栖控制面板：

  • 配置面板：支持动态机动切换 LLM 模型（包含自定义模型填入）、双渠道推送（企业微信 Webhook + Server酱 SendKey 修改）。
  • 情报仪表盘：直观展示最近 20 条推送记录的状态与简述，告别翻阅上千行控制台日志的痛苦。
  • 偏好设置：提供 8 大校园分类（讲座/竞赛/考试/放假等）的多选开关。

💡 视觉去 Emoji 化整改经验

早期 AI 在写 Gradio 界面时，极其喜欢在 label 里塞满 🤖、🛠️、📅 等原生 Emoji，界面充满了廉价的机器人感。

为了提升质感，我们实施了全面视觉净化：

  1. 彻底移除所有粗暴的系统内置 Emoji。
  2. 利用 Layout 权重：在 gr.Row() 中严格控制 scale 比例（例如：API Key 输入框设为 scale=3，启用开关设为 scale=1），彻底拉正了被 AI 挤压变形的歪斜输入框。
  3. 区块化设计：使用 gr.Group() 将功能相近的配置包裹，形成轻量线框感，整体风格向 GitHub Primer 靠拢。

0x05 后记与未来展望

截至目前，Academic-information-report-agent 已经在本地稳定运行。每天清晨，它都会默默地启动无头浏览器，像一个冷酷的特工一样在各个校园平台搜刮情报，经过大模型的精炼后，将最新、最干净的简报精准推送到我的手机上。

从一开始被 504 错误折磨，到后来和 SQLite 死锁斗智斗勇，这个项目让我深刻体会到：开发一个实用的 AI Agent，核心往往不在于 Prompt 写得多漂亮，而在于数据流的处理、异常边界的防御以及底层工程的闭环。

下一步，我计划加入基于本地向量库的“特定导师课题组动态追踪”，让它变得更加硬核。Stay tuned！
